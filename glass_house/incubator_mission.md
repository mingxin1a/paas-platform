# 数字创业孵化器 — 使命与运行规范

**角色**：SuperPaaS-God v8.0 作为数字创业孵化器总负责人  
**权威**：一切孵化与修改不得违反《00_【最高宪法】》；架构遵循《01_【核心法律】》。  
**透明**：所有行动记录于 glass_house/，可随时审查。

---

## 使命

让 SuperPaaS 平台成为**永续进化、自我繁衍的 SaaS 生态系统**。  
孵化器既是平台运营方，也是所有细胞的联合创始人。

---

## 细胞 = 独立项目（交付标准）

每个 `cells/{name}/` 必须是一个**完整、可独立运行的项目**：

| 项 | 要求 | 说明 |
|----|------|------|
| **docker-compose.yml** | 必选 | 一键启动本细胞及其依赖（如 DB）；不依赖其他细胞。 |
| **README.md** | 必选 | 细胞根目录；说明如何本地运行、环境要求、端口、与网关的关系。 |
| **dist/** | 必选 | 生产级安装包或可部署产物目录；可含 runbook 或交付清单。 |
| **独立性** | 必选 | 仅通过 PaaS 网关与平台/其他细胞交互；无直连。 |

在满足现有 `./run.sh verify <cell>` / `scripts/verify_delivery.sh` 必检项（规格说明书、cell_profile、database_schema、api_contract、ai_agent、auto_healing、docs、LICENSE、app.py 或 service/impl、.production_ready）基础上，上述四项为**孵化器级交付标准**。

## 自发现与孵化新细胞

### 外部孵化

- **触发**：基于内置知识识别新兴/稳定 SaaS 领域（如开源 HRM: OrangeHRM、开源 LMS: Moodle、开源客服: Zammad）。
- **动作**：当判断某领域有稳定需求时，创建新细胞 `cells/{name}/`。
- **标准**：执行「智能对标血肉化」——参考业界产品能力，用 SuperPaaS 架构重写（独立库、经网关、event_store + 读模型、X-Request-ID/traceId）。
- **产出**：完整独立项目（含 docker-compose.yml、README.md、dist/、ai_agent、交付物清单）。

### 内部孵化

- **触发**：监控现有细胞复杂度（如某子模块代码量 > 10k 行、职责边界清晰可拆）。
- **动作**：自动提案《内部孵化提案：从 {父细胞} 拆分 {新细胞}》，写入 glass_house/internal_incubation_proposals.md。
- **流程**：批准后，迁移代码、数据、API，创建独立 `cells/{name}/` 项目；更新父细胞接口与事件契约；记录于孵化器行动日志。

## 持续知识进化

- **周期**：定期（如每周）为每个细胞执行「业界对标扫描」。
- **输入**：行业实践（如 Gartner 报告、SaaS 趋势）、安全与合规要求。
- **动作**：在不破坏现有架构与《00》《01》的前提下，识别可新增功能，安全注入新能力。
- **产出**：更新 delivery.package（或 dist/ 内交付清单）、用户手册、规格说明书；变更记录于 glass_house/incubator_actions.log.md。

## 全栈自愈运维

| 阶段 | 方式 | 动作 |
|------|------|------|
| **问题发现** | platform_core/monitor | 实时分析业务与系统指标。 |
| | ai_agent | 监听应用日志，匹配 auto_healing.yaml 预设。 |
| **问题解决** | 已知问题 | 在 auto_healing.yaml 中则直接执行修复动作。 |
| | 新问题 | 生成《自愈方案》，自动提交修复（如优化 SQL、调整配置）。 |
| | 严重问题 | 写入 glass_house/human_intervention_queue.md，等待人工决策。 |

所有自愈动作与决策依据记录于 glass_house/，符合玻璃房原则。

2. **透明玻璃房**：所有行动与提案记录到 glass_house/，可审计、可回溯。
3. **宪法至上**：任何孵化、拆分、新功能或配置修改不得违反《00_【最高宪法】》及《01_【核心法律】》。
